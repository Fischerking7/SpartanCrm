You are a senior full-stack engineer building a Finance/Accounting overhaul for our Postgres-backed CRM in Replit.

CONTEXT
- We no longer export to QuickBooks.
- Customers pay the bank directly (ACH). We do NOT pay reps through the app.
- We receive client-provided “sales/acceptance” documents (remittance-like but actually acceptance/production), like:
  Columns: Customer Name, Service Type, Utility, Client, Date Sold, Status, Usage, Rate, Rejection Reason
  Example Status values: Enrolled, Rejected (and potentially Pending)
- This file is NOT proof of cash receipt. It is proof of client acceptance/production and should create expected receivables (AR expectations).
- Later we may ingest separate payment/remittance or bank deposit data to mark items paid, but that is NOT required for MVP.

GOAL
Replace the existing “Accounting” tab (currently QuickBooks export/import) with a workflow-driven “Remittance & AR” module:
1) Import client sales/acceptance file
2) Map + validate columns (save mapping per client)
3) Auto-match rows to existing Orders
4) Post “Client Accepted” status to orders + create AR Expectations for Enrolled rows
5) Provide exception queue for unmatched/ambiguous rows
6) Provide reports for Rep/Team/Global: Enrolled (accepted) totals by week/month/YTD and Rejected counts/reasons
7) Provide AR view: open expected receivables by client + aging

TECH REQUIREMENTS
- Use Postgres. Add migrations.
- Use idempotency/dedupe: file_hash to block duplicate imports; row_fingerprint to dedupe rows within import.
- Audit trail: store created_by, created_at, and maintain immutable raw rows.
- Locking: after an import is posted, prevent edits to normalized rows except via explicit “Unpost/Repost” admin action (or create adjustments later; for MVP allow unpost if not locked).
- Performance: include basic indexes for matching (customer name normalized, sale date, client_id).

- For Rejected rows, set order acceptance status to REJECTED if matched; store rejection reason in an order_notes/audit table or on the finance row.

MATCHING LOGIC (AUTO)
Implement an auto-matcher that runs after normalization:
Given a finance_import_row:
Score candidate orders where:
- Same client_id (or client name matches) AND
- order_date within +/- 7 days of sale_date (configurable) AND
- customer_name_norm similarity (exact contains match) AND
- service_type matches order.service_type if available
Compute match_confidence:
- +60 if customer_name_norm exact match
- +40 if customer_name_norm contains or contained-by
- +20 if service_type match
- +20 if sale_date within +/- 2 days
- +10 if expected_amount matches order.expected_amount_cents (if present)
Pick best candidate.
If best_confidence >= 85 and unique => MATCHED
If multiple candidates within 10 points => AMBIGUOUS
Else => UNMATCHED
Store match_reason jsonb with scores and candidate order_ids.

POSTING RULES
When user clicks “Post Import”:
- Require import status MATCHED (or allow with unresolved exceptions but warn)
- For each normalized row:
  - If Status == Enrolled AND match_status == MATCHED:
      - Mark the order as client accepted (client_acceptance_status=ACCEPTED, client_accepted_at=now)
      - Create ar_expectation if not exists (unique finance_import_row_id)
  - If Status == Rejected AND match_status == MATCHED:
      - Mark order acceptance status REJECTED (optional for MVP)
- Set finance_import.status = POSTED
- Optionally LOCK after manager confirmation.

VALIDATION & DEDUPE
- Compute file_hash (sha256) from file bytes; enforce uniqueness per client unless “force reimport” admin.
- Compute row_fingerprint = sha256(client_id + customer_name_norm + sale_date + expected_amount_cents + service_type)
- If duplicate fingerprint in same import, mark as duplicate and skip or flag.
