Extend the Iron Crest commission structure with two additions:
(1) a new Accounting override tier, and (2) an approval workflow 
for all override earnings before they become payable.

---

### TASK 1B — Add ACCOUNTING to override recipient logic

In the generateOverrideEarnings function (called after order approval),
add a third override recipient lookup for ACCOUNTING role users:

Current flow handles:
  1. EXECUTIVE recipients (Director override)
  2. OPERATIONS/ADMIN recipients (Admin override)

Add:
  3. ACCOUNTING recipients — users with role 'ACCOUNTING' who are 
     in the rep's org hierarchy or company-wide (ACCOUNTING is a 
     back-office role, not a field hierarchy role, so all active 
     ACCOUNTING users receive the override when accountingOverrideCents > 0)

Logic to add in generateOverrideEarnings:

  const accountingOverrideAmount = rateCard.accountingOverrideCents 
                                    ? rateCard.accountingOverrideCents / 100 
                                    : 0;

  if (accountingOverrideAmount > 0) {
    const accountingUsers = await storage.getUsersByRole("ACCOUNTING");
    for (const accountingUser of accountingUsers) {
      await storage.createOverrideEarning({
        salesOrderId: order.id,
        recipientUserId: accountingUser.id,
        recipientRole: "ACCOUNTING",
        overrideType: "ACCOUNTING_OVERRIDE",
        amount: (accountingOverrideAmount / accountingUsers.length).toFixed(2),
        // Split evenly among all active ACCOUNTING users
        // If only one ACCOUNTING user exists, they receive the full amount
        status: "PENDING_APPROVAL",
        payRunId: null,
      });
    }
  }

NOTE: If your business has exactly one ACCOUNTING user, the split 
logic still works correctly — they receive 100%. If the number of 
ACCOUNTING users is 0, skip creation and log a rateIssue with 
type "MISSING_ACCOUNTING_RECIPIENT".

Update the overrideEarnings table to support the new overrideType 
value "ACCOUNTING_OVERRIDE" if it uses an enum — add it to the 
enum definition and create a Drizzle migration.

### PART 2 — OVERRIDE APPROVAL WORKFLOW

Override earnings must go through an approval step before they 
become payable and includable in a pay run. This applies to ALL 
override types: Director, Admin, and Accounting.

---

### TASK 2A — Add approval status to overrideEarnings

Add the following columns to the overrideEarnings table 
(Drizzle migration):

  - approvalStatus: varchar(20) not null default 'PENDING_APPROVAL'
    Allowed values: 'PENDING_APPROVAL' | 'APPROVED' | 'REJECTED' | 'VOIDED'
  - approvedByUserId: varchar references users(id) nullable
  - approvedAt: timestamp nullable
  - rejectedByUserId: varchar references users(id) nullable  
  - rejectedAt: timestamp nullable
  - rejectionReason: text nullable
  - approvalNote: text nullable

All newly created overrideEarnings records must be created with 
approvalStatus = 'PENDING_APPROVAL' regardless of override type.

Update the existing createOverrideEarning storage method to default 
approvalStatus to 'PENDING_APPROVAL' if not explicitly provided.

### TASK 2B — Override approval API endpoints

Add the following endpoints, all protected by auth middleware:

--- 

GET /api/admin/override-earnings/pending
  Access: EXECUTIVE and OPERATIONS roles only
  
  Returns all overrideEarnings records where:
    approvalStatus = 'PENDING_APPROVAL'
  
  Enriched response per record:
  {
    id: string,
    salesOrderId: string,
    orderInvoiceNumber: string,
    orderCustomerName: string,
    orderDateSold: string,
    recipientUserId: string,
    recipientName: string,
    recipientRole: string,
    overrideType: string,
    amount: string,
    approvalStatus: string,
    createdAt: string,
    order: {
      repId: string,
      repName: string,
      repRole: string,
      serviceName: string,
      providerName: string,
      jobStatus: string,
      approvalStatus: string,
    }
  }

  Query parameters:
    - overrideType: filter by 'DIRECTOR_OVERRIDE' | 'ADMIN_OVERRIDE' 
                    | 'ACCOUNTING_OVERRIDE'
    - recipientUserId: filter to a specific recipient
    - orderId: filter to a specific order

---

GET /api/admin/override-earnings/pending/count
  Access: EXECUTIVE and OPERATIONS roles only
  Returns: { count: number }
  Used for badge counts in the UI navigation.

---

POST /api/admin/override-earnings/:id/approve
  Access: 
    - EXECUTIVE role can approve DIRECTOR_OVERRIDE and ACCOUNTING_OVERRIDE
    - OPERATIONS role can approve ADMIN_OVERRIDE and ACCOUNTING_OVERRIDE
    - Neither role can approve their own override earnings 
      (if the approver is also a recipient, reject with 403)
  
  Request body:
    { note?: string }
  
  Actions:
    1. Validate the user has permission to approve this override type
    2. Validate the approver is not the recipient
    3. Set approvalStatus = 'APPROVED'
    4. Set approvedByUserId = req.user.id
    5. Set approvedAt = now()
    6. Create an audit log entry:
       action: 'OVERRIDE_EARNING_APPROVED'
       recordId: overrideEarning.id
       afterJson: { approvedBy, overrideType, amount, recipientName }
    7. If the linked order has a payRunId already assigned, 
       do NOT auto-include — leave payRunId null and let the 
       next pay run pick it up
    8. Return the updated overrideEarning record

---

POST /api/admin/override-earnings/:id/reject
  Access: Same role rules as approve endpoint
  
  Request body:
    { reason: string }  (required — rejection must have a reason)
  
  Actions:
    1. Validate permissions (same as approve)
    2. Validate approver is not recipient
    3. Validate reason is provided and non-empty
    4. Set approvalStatus = 'REJECTED'
    5. Set rejectedByUserId = req.user.id
    6. Set rejectedAt = now()
    7. Set rejectionReason = reason
    8. Create audit log:
       action: 'OVERRIDE_EARNING_REJECTED'
       afterJson: { rejectedBy, reason, overrideType, amount, recipientName }
    9. Return the updated overrideEarning record

---

POST /api/admin/override-earnings/bulk-approve
  Access: EXECUTIVE and OPERATIONS roles only
  
  Request body:
    { ids: string[], note?: string }
  
  Actions:
    1. Filter the provided IDs to only those the requesting user 
       has permission to approve (by overrideType and not self)
    2. Bulk approve all qualifying records
    3. Skip and collect IDs that cannot be approved (wrong type, 
       self-approval, already approved)
    4. Create a single audit log entry for the bulk action
    5. Return:
       {
         approved: number,
         skipped: number,
         skippedIds: string[],
         skippedReasons: Record<string, string>
       }

### TASK 2D — Notification triggers for override approval

  - For DIRECTOR_OVERRIDE: notify all EXECUTIVE users
  - For ADMIN_OVERRIDE: notify all OPERATIONS users

    notificationType: "OVERRIDE_PENDING_APPROVAL",
    subject: "Override Earning Pending Approval",
    body: `A ${overrideType} override of $${amount} for order 
           ${invoiceNumber} (${customerName}) is pending your approval.`,
    userId: notifyUser.id,
    status: "PENDING",
    isRead: false,
  }

Use the existing storage.createEmailNotification method for this.

When an override is approved or rejected, notify the RECIPIENT:
  
  Approved notification to recipient:
    subject: "Override Earning Approved"

  Rejected notification to recipient:
    subject: "Override Earning Rejected"  
    body: `Your ${overrideType} override of $${amount} for order 
           ${invoiceNumber} was rejected. Reason: ${rejectionReason}`
