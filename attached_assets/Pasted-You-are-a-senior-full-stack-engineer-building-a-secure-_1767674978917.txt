You are a senior full-stack engineer building a secure Sales Ops + commissions + QuickBooks reconciliation app designed to scale from 5 reps to 50–200 reps quickly.

Build a Replit-ready monorepo with:
- apps/server (Node.js + Express + Prisma + Postgres)
- apps/web (React + Vite)
TypeScript end-to-end.

Auth:
- RepID + password (bcrypt hash)
- JWT sessions
Roles: REP, MANAGER, ADMIN
Strict access control (enforce on ALL endpoints, exports, imports):
- REP: only records where repId = their repId
- MANAGER: only records for reps assigned to their team
- ADMIN: full access

Core definitions (must be consistent everywhere):
- Earned = jobStatus=COMPLETED AND approvalStatus=APPROVED
- Paid = imported from QuickBooks (paidDate exists)
- Chargebacks = negative entries tied to invoiceNumber; never modify original order
- Incentives and upline overrides are separate from base commission; do not rewrite history

=== SCALE ADDITIONS (NEW) ===
1) Add AuditLog table: records who changed what and when (action, tableName, recordId, beforeJson, afterJson, userId, createdAt).
Log critical actions: login (optional), order create/update, approval/reject, invoice creation, export batch, payment import, chargeback import/manual, adjustment create/approve/reject, rate card changes, user deactivation.

2) Add PayRun table (weekly payouts, no forecasting):
- PayRun: id, weekEndingDate, status(DRAFT/FINALIZED), createdByUserId, createdAt, finalizedAt
- Imported payments and chargebacks must link to a PayRun when possible (payRunId nullable).
Provide Admin UI to create/select current PayRun and import payment CSV into that pay run.
Rep/Manager dashboards should show "Paid this week" based on PayRun or derived from paidDate.

3) Add Exception Queues (tables + endpoints + UI):
- UnmatchedPayment: id, payRunId(optional), rawRowJson, reason, createdAt, resolvedByUserId(optional), resolvedAt(optional), resolutionNote(optional)
- UnmatchedChargeback: same structure
- RateIssue: id, salesOrderId, type(MISSING_RATE/CONFLICT_RATE), details, createdAt, resolvedAt(optional), resolvedByUserId(optional)
- ApprovalAging view/query (not necessarily table): show completed+unapproved older than 24h/48h.

4) Bulk Admin Actions:
- Bulk approve orders from approvals queue (with guardrails: skip if missing rate or conflicts)
- Bulk export approved jobs (creates ExportBatch)
- Bulk mark exported (only via export action)
- Bulk resolve unmatched rows after manual mapping

5) Immutability rules:
- After approvalStatus=APPROVED, lock key order fields (client/provider/service/installDate/commission values/invoice number).
- Corrections must be done via Adjustments (manual inputs) or via explicit Admin "Correction" action that logs in AuditLog and creates an Adjustment automatically.
No silent edits on approved orders.

6) Reference data governance:
Provider/Client/Service must be lookup tables only (no free text).
Support deactivate instead of delete. Prevent using inactive items in new orders.

=== BUSINESS SERVICES ===
1) Invoice sequencing:
Use Counter table or transaction-safe mechanism:
- Counter { key: "invoice", value int }
On approval, in a DB transaction:
- increment counter
- set invoiceNumber like INV-YYYY-000001

2) Commission calculation service:
Deterministic rate card selection priority:
- Match provider + service + conditions + effective date + active
- Prefer client-specific rule over client=ANY
- Validate lines range if present
If no match -> create RateIssue(MISSING_RATE)
If multiple matches -> create RateIssue(CONFLICT_RATE)
Only calculate baseCommissionEarned when there is a single valid match.
Calculate incentives separately.
Allow admin manual override before approval.

3) Locking after approval:
If approved, block updates to locked fields (except admin correction flow that logs and creates adjustment).

4) Net totals:
Paid totals are driven by imported payments + chargebacks (negative).
Adjustments add/subtract from earned totals when approved.
