Using the complete existing infrastructure of the Iron Crest application,
implement the following automation, streamlining, and simplification 
improvements across every module. The guiding principle for every change:

  HUMANS SHOULD ONLY SEE EXCEPTIONS.
  EVERYTHING ROUTINE SHOULD BE INVISIBLE AND AUTOMATIC.

SECTION 1 — INTELLIGENT EXCEPTION MANAGEMENT
===================================================================

The single most important UX improvement is an exceptions-first 
admin interface. Instead of admins reviewing tables of data, the 
system surfaces only what needs attention.

### TASK 1A — Unified exception queue

Create a new endpoint:
GET /api/admin/exceptions/queue
Access: ADMIN, OPERATIONS

This endpoint aggregates ALL pending actions across every module 
into a single prioritized list. This becomes the admin's homepage.

Response:
  {
    urgent: ExceptionItem[],      // must act today
    high: ExceptionItem[],        // act within 3 days  
    medium: ExceptionItem[],      // act within 7 days
    low: ExceptionItem[],         // informational
    totalCount: number,
    lastRefreshedAt: string,
  }

ExceptionItem shape:
  {
    id: string,                   // unique exception ID
    type: ExceptionType,
    priority: 'URGENT' | 'HIGH' | 'MEDIUM' | 'LOW',
    title: string,                // human readable one-liner
    description: string,          // what needs to happen
    affectedEntityId: string,     // orderId, repId, arId, etc.
    affectedEntityType: string,
    actionUrl: string,            // deep link to the relevant page
    actionLabel: string,          // button text e.g. "Review Order"
    createdAt: string,
    staleSince?: string,          // how long this has been waiting
    estimatedImpactCents?: number // dollar value at stake
  }

ExceptionType values and their priority rules:

  URGENT (act today):
    'ORDER_APPROVAL_OVERDUE'      
      // Completed order unapproved for 3+ days
      // estimatedImpact: baseCommissionEarned
    'NEGATIVE_MARGIN_DETECTED'    
      // ironCrestProfitCents < 0 on any order
    'PAY_RUN_FINALIZATION_DUE'    
      // Pay run in DRAFT status past its scheduled pay date
    'CHARGEBACK_REQUIRES_ACTION'  
      // Chargeback applied with no response in 48 hours
    'AR_PAYMENT_OVERDUE'          
      // AR expectation open 45+ days past expectedFromDate

  HIGH (3 days):
    'OVERRIDE_PENDING_APPROVAL'   
      // Override earning waiting >24 hours
    'DISPUTE_UNRESPONDED'         
      // Commission dispute with no admin response in 48 hours
    'ADVANCE_OVERDUE'             
      // Advance in REPAYING status with no payment in 2 periods
    'AR_VARIANCE_LARGE'           
      // AR variance > $50 unresolved
    'RATE_CARD_MISSING'           
      // Approved order has no rate card applied

  MEDIUM (7 days):
    'OVERRIDE_PENDING_APPROVAL'   
      // Override waiting <24 hours (lower priority version)
    'REP_MISSING_TAX_PROFILE'     
      // Rep in active pay run with no tax profile
    'AR_PARTIAL_PAYMENT'          
      // AR partially paid, balance outstanding 14+ days
    'FINANCE_IMPORT_UNMATCHED'    
      // Finance import has unmatched rows 7+ days old
    'ADVANCE_AWAITING_APPROVAL'   
      // Advance request pending >3 days

  LOW (informational):
    'STALE_AR_NO_EXPECTATION'     
      // Approved order 30+ days with no AR expectation
    'REP_NO_SALES_7_DAYS'         
      // Active rep with no orders in 7 days (field context)
    'RATE_CARD_EXPIRING_SOON'     
      // Rate card effectiveEnd within 14 days

Build this by querying the relevant tables for each exception type.
Cache the result for 5 minutes using a simple in-memory cache 
(store last result + timestamp on the module, recompute if stale).

SECTION 2 — SMART AUTO-APPROVAL RULES
===================================================================

### CONCEPT
Routine approvals that meet all quality criteria should not require 
human review. Humans should only approve exceptions.

### TASK 2A — Auto-approval rule engine

Create server/autoApprovalEngine.ts

Export:
  async function evaluateOrderForAutoApproval(
    orderId: string
  ): Promise<{
    eligible: boolean,
    confidence: number,       // 0-100
    reasons: string[],        // why eligible or not
    blockers: string[],       // what prevents auto-approval
  }>

Auto-approval criteria (ALL must be true):
  1. jobStatus = 'COMPLETED'
  2. approvalStatus = 'UNAPPROVED'
  3. A valid rate card was found and applied (appliedRateCardId not null)
  4. baseCommissionEarned > 0
  5. ironCrestProfitCents >= 0 (no negative margin)
  6. No active commission dispute linked to this order
  7. No active chargeback linked to this order
  8. The rep has no isPayrollHeld flag
  9. The AR expectation for this order (if exists) is not WRITTEN_OFF
  10. The order's invoiceNumber is unique (no duplicates)
  11. customerName is not blank
  12. dateSold is within the last 180 days

Confidence scoring:
  Start at 100, subtract for each soft concern:
  - No AR expectation exists: -15
  - Rep has had a chargeback in the last 90 days: -10
  - Order amount is >2x this rep's average order: -20
  - dateSold is a weekend: -5 (unusual for telecom)
  - No installDate set: -10

  If confidence >= 80 AND all hard criteria pass: eligible = true
  If confidence < 80: eligible = false, flag for human review

### TASK 2B — Auto-approval scheduler job

Extend the existing scheduler with a new job that runs every 
15 minutes:

  async function runAutoApprovalSweep(): Promise<void>

  1. Fetch all orders where:
     jobStatus = 'COMPLETED'
     AND approvalStatus = 'UNAPPROVED'
     AND autoApprovalAttemptedAt IS NULL (new field, see below)
  
  2. For each order, call evaluateOrderForAutoApproval()
  
  3. If eligible:
     a. Approve the order (set approvalStatus = 'APPROVED',
        approvedByUserId = system user ID, approvedAt = now())
     b. Call generateOverrideEarnings for the order
     c. Set payrollReadyAt if AR is already satisfied
     d. Create audit log: 'AUTO_APPROVAL_SYSTEM'
        afterJson: { confidence, reasons }
     e. Set autoApprovalAttemptedAt = now(), 
        autoApprovalResult = 'APPROVED'
  
  4. If not eligible:
     a. Set autoApprovalAttemptedAt = now()
     b. Set autoApprovalResult = 'FLAGGED'
     c. Set autoApprovalBlockers = JSON.stringify(blockers)
     d. Create exception in queue: ORDER_APPROVAL_OVERDUE 
        with description of blockers
  
  Add to salesOrders table (migration):
    - autoApprovalAttemptedAt: timestamp nullable
    - autoApprovalResult: varchar(20) nullable
      Values: 'APPROVED' | 'FLAGGED' | 'SKIPPED'
    - autoApprovalBlockers: text nullable (JSON array of reasons)
    - autoApprovedBySystem: boolean default false

### TASK 7A — Duplicate order detection

Add a duplicate detection check to the order creation endpoint.
Before creating an order, check:

  const potentialDuplicate = await db.select()
    .from(salesOrders)
    .where(and(
      eq(salesOrders.repId, data.repId),
      ilike(salesOrders.customerName, data.customerName),
      gte(salesOrders.dateSold, thirtyDaysAgo),
      ne(salesOrders.approvalStatus, 'REJECTED')
    ))

If a potential duplicate is found:
  - Do not block the creation
  - Set a flag on the new order: 
    isDuplicateFlag: boolean default false
    duplicateFlagReason: text nullable
  - Return a warning in the API response:
    { order: newOrder, warning: { 
      type: 'POTENTIAL_DUPLICATE',
      existingOrderId: string,
      existingInvoiceNumber: string,
      message: string
    }}

Add to salesOrders table (migration):
  - isDuplicateFlag: boolean default false
  - duplicateFlagReason: text nullable
  - duplicateOfOrderId: varchar nullable

### TASK 7B — Commission calculation integrity check

Add a daily background job: commissionIntegrityCheck

For all orders approved in the last 30 days:
  1. Re-evaluate the rate card that should apply
  2. Compare to the stored baseCommissionEarned
  3. If difference > $1.00: create a rateIssue record
     type: 'COMMISSION_MISMATCH'
     details: JSON with expected vs actual amounts
  4. Add to exception queue: 'RATE_CARD_MISSING' or 
     create a new type 'COMMISSION_MISMATCH'

This catches cases where rate cards were updated after 
orders were approved.

### TASK 7C — Orphaned record cleanup

Add a weekly background job: orphanedRecordCheck

Checks for data integrity issues:
  1. Override earnings with no linked salesOrder
  2. Pay statement line items with no linked payStatement
  3. AR expectations with no linked order
  4. Finance import rows marked MATCHED but matchedOrderId is null
  5. Orders in a payRun where the payRun has been deleted

For each issue found: create a LOW priority exception in the queue.
Do not auto-delete anything — just flag for admin review.

GENERAL CONSTRAINTS
===================================================================

- Create ONE consolidated Drizzle migration for all schema changes
- Do not modify existing API response shapes — only ADD fields
- Do not touch the frontend
- All new scheduled jobs must integrate with the existing 
  scheduler infrastructure and log to backgroundJobs table
- Exception queue must be computed from live data, not a 
  separate stored queue (except for dismissals)
- All monetary values stored as INTEGER cents
- Auto-approval system must create detailed audit logs so every 
  system-approved order is fully traceable
- System settings must be validated on write — unknown keys 
  should be rejected with 400
- The automation rules engine must fail gracefully — a broken 
  rule should log to backgroundJobs and continue, never crash
- Every new storage method must have JSDoc comments
- Self-approval prohibition must be enforced everywhere
- Add integration points (comments) in the code wherever a 
  future webhook or external integration would connect — label 
  these: // INTEGRATION_POINT: {description}
