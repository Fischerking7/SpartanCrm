Integrate the Iron Crest contractor onboarding system directly 
into the existing Iron Crest application. This is not a separate 
app — it is a new module within the existing codebase.

The onboarding system handles 1099 independent contractor reps 
completing 6 legal documents before gaining app access. It must 
connect bidirectionally with the existing user, payroll, and 
compliance infrastructure.

Governing law: State of New Jersey
Ops email: ironcrestoperations@ironcrestai.com
Onboarding URL: https://onboarding.ironcrestsolutions.com

### TASK 2B — OTP generation and sending

Create server/onboarding/otpService.ts:

  import crypto from 'crypto';
  import bcrypt from 'bcryptjs';

  export async function generateAndSendOtp(
    userId: string,
    phoneNumber: string,
    repName: string
  ): Promise<{ success: boolean; error?: string }>

  Steps:
  1. Generate 6-digit OTP:
     const otp = crypto.randomInt(100000, 999999).toString();
  
  2. Hash with bcrypt cost 10:
     const otpHash = await bcrypt.hash(otp, 10);
  
  3. Store on user record:
     await storage.updateUser(userId, {
       onboardingOtpHash: otpHash,
       onboardingOtpExpiresAt: new Date(Date.now() + 48 * 60 * 60 * 1000),
       onboardingOtpAttempts: 0,
       onboardingOtpLockedAt: null,
       onboardingStatus: 'OTP_SENT',
     });
  
  4. Send via Twilio SMS:
     const client = twilio(
       process.env.TWILIO_ACCOUNT_SID,
       process.env.TWILIO_AUTH_TOKEN
     );
     await client.messages.create({
       body: `Your Iron Crest onboarding code is: ${otp}. 
              Valid for 48 hours. 
              Visit: ${process.env.FRONTEND_URL}/onboarding`,
       from: process.env.TWILIO_FROM_NUMBER,
       to: phoneNumber,
     });
  
  5. Log to onboarding_audit_log:
     action: 'OTP_GENERATED' then 'OTP_SENT_SMS'
     detail: { repName, phoneLastFour: phone.slice(-4) }
  
  6. Return success

### TASK 2C — OTP verification

Create server/onboarding/verifyOtp.ts:

  export async function verifyOtp(
    userId: string,
    submittedOtp: string,
    ipAddress: string,
    userAgent: string
  ): Promise<{ 
    success: boolean; 
    token?: string; 
    error?: string;
    locked?: boolean;
  }>

  Steps:
  1. Load user record
  
  2. Check if locked:
     if (user.onboardingOtpLockedAt) {
       log: 'OTP_FAILED_ATTEMPT' with locked: true
       return { success: false, locked: true, 
                error: 'Account locked. Contact your manager.' }
     }
  
  3. Check expiry:
     if (new Date() > user.onboardingOtpExpiresAt) {
       return { success: false, 
                error: 'Code expired. Ask your manager to resend.' }
     }
  
  4. Verify hash:
     const valid = await bcrypt.compare(
       submittedOtp, 
       user.onboardingOtpHash
     );
  
  5a. If valid:
     -- Clear OTP fields (single use)
     await storage.updateUser(userId, {
       onboardingOtpHash: null,
       onboardingOtpExpiresAt: null,
       onboardingOtpAttempts: 0,
       onboardingStatus: 'OTP_VERIFIED',
       onboardingStartedAt: new Date(),
     });
     
     -- Issue JWT (4-hour expiry, stored in memory not localStorage)
     const token = jwt.sign(
       { userId, purpose: 'onboarding' },
       process.env.JWT_SECRET,
       { expiresIn: '4h' }
     );
     
     -- Log success
     log: 'OTP_VERIFIED' with ipAddress, userAgent
     
     return { success: true, token }
  
  5b. If invalid:
     const attempts = user.onboardingOtpAttempts + 1;
     
     if (attempts >= 5) {
       -- Lock the account
       await storage.updateUser(userId, {
         onboardingOtpAttempts: attempts,
         onboardingOtpLockedAt: new Date(),
         onboardingStatus: 'OTP_SENT',
       });
       
       -- Notify manager via in-app notification
       if (user.assignedManagerId) {
         await storage.createEmailNotification({
           userId: user.assignedManagerId,
           notificationType: 'ONBOARDING_OTP_LOCKED',
           subject: 'Onboarding Access Locked',
           body: `${repName}'s onboarding access has been 
                  locked after 5 failed attempts. 
                  Please generate a new OTP from the rep 
                  management screen.`,
           status: 'PENDING',
           isRead: false,
         });
       }
       
       log: 'OTP_LOCKED'
       return { success: false, locked: true,
                error: 'Too many attempts. Contact your manager.' }
     }
     
     await storage.updateUser(userId, {
       onboardingOtpAttempts: attempts,
     });
     
     log: 'OTP_FAILED_ATTEMPT' with { attempts }
     return { success: false, 
              error: `Invalid code. ${5 - attempts} attempts remaining.` }

Access: Public (no auth — this is the entry point)

Request body:
  { repId: string, otp: string }

not the internal UUID. This is what the rep knows about themselves.

Actions:
  1. Look up user by repId
  2. Verify user has onboardingStatus in 
     ['OTP_SENT', 'OTP_VERIFIED', 'IN_PROGRESS']
  3. Extract IP from request:
     const ip = req.headers['x-forwarded-for'] || 
                 req.socket.remoteAddress
  4. Call verifyOtp(user.id, otp, ip, userAgent)
  5. Return token or error

Rate limiting: max 10 requests per IP per hour

SECTION 6 — ONBOARDING STATUS IN APP
===================================================================

### TASK 6A — Block app access for incomplete onboarding

Add to the main auth middleware (the existing auth function):

After validating the JWT:
  -- For REP, LEAD, MANAGER roles only:
  if (['REP', 'LEAD', 'MANAGER'].includes(user.role)) {
    -- Check if onboarding is complete
    if (user.onboardingStatus !== 'APPROVED' 
        && user.appAccessGrantedAt === null) {
      -- Allow access only to onboarding routes
      const allowedPaths = [
        '/api/onboarding/',
        '/api/my/profile',
        '/api/notifications',
      ];
      const isAllowed = allowedPaths.some(p => 
        req.path.startsWith(p)
      );
      if (!isAllowed) {
        return res.status(403).json({
          message: 'Onboarding not complete',
          onboardingStatus: user.onboardingStatus,
          redirectTo: '/onboarding',
        });
      }
    }
  }

This means a new rep who hasn't completed onboarding 
gets a 403 on every endpoint except the onboarding ones.
The frontend uses the redirectTo field to send them 
to the onboarding portal.

### TASK 6B — Onboarding status in user responses

Add onboardingStatus and appAccessGrantedAt to the 
user object returned by GET /api/auth/me 
(or wherever the current user info is fetched).

The frontend uses this to:
  - Show an onboarding banner if not yet complete
  - Show a pending banner if submitted but not yet approved
  - Show normal app if approved

### TASK 6C — Onboarding progress in rep management

In the OPERATIONS rep management screen data 
(GET /api/admin/users or rep listing endpoint):

Add to each rep's data:
  onboardingStatus: string
  onboardingSubmittedAt: timestamp | null
  backgroundCheckStatus: string
  drugTestStatus: string

Add to the exception queue (OPERATIONS type):
  'ONBOARDING_OVERDUE':
    Priority: HIGH
    Trigger: User created 7+ days ago with 
             onboardingStatus not in ['SUBMITTED', 'APPROVED']
    Title: "{n} reps have not completed onboarding"
    estimatedImpact: null (compliance risk, not dollar risk)
    Action: links to rep management filtered to 
            onboardingStatus = 'NOT_STARTED' or 'OTP_SENT'

### TASK 6D — Manager visibility into team onboarding

Add to GET /api/my/team-health response:
  onboardingMetrics: {
    totalReps: number,
    onboardingComplete: number,
    onboardingPending: number,      // submitted, not approved
    onboardingIncomplete: number,   // not yet submitted
    complianceClear: number,        // both checks cleared
  }

MANAGER and DIRECTOR can see this for their scope.
EXECUTIVE sees company-wide.

===================================================================
GENERAL CONSTRAINTS
===================================================================

- Create ONE consolidated Drizzle migration for all changes
- The onboarding_audit_log must be truly append-only:
  No updateOnboardingAuditLog or deleteOnboardingAuditLog 
  methods in storage — only createOnboardingAuditLog
- SSN and financial data must use the SAME encryption 
  utilities already used in the MDU module (encryptSsn, 
  decryptSsn) for consistency
- Never log SSN, routing numbers, or account numbers 
  anywhere — not in console, not in audit logs, not in 
  error messages
- Rate limit the OTP verification endpoint:
  10 attempts per IP per hour
- OTP is never stored in plaintext anywhere — only bcrypt hash
- JWT for onboarding uses purpose: 'onboarding' claim and 
  must be rejected by the main auth middleware
- All S3 uploads must use private ACL + server-side encryption
- Pre-signed S3 URLs must have maximum 15-minute expiry 
  when served via API, 7-day expiry in emails
- Do not touch the frontend
- All new storage methods must have JSDoc comments
- Add ONBOARDING_SUBMITTED and ONBOARDING_APPROVED 
  to the exception queue as MEDIUM priority items
- The onboarding flow must work even if Twilio is 
  unavailable — fall back gracefully and log the failure,
  do not crash the user creation flow
- Phone number validation: normalize to E.164 format 
  before storing and before sending SMS
